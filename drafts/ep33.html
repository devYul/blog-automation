← 32편: GitHub Actions로 블로그 완전 자동화

지난달이었다. 블로그를 시작하고 처음 받아보는 이메일 제목 — "Google 애드센스: 사이트 검토 결과". 기대 반 불안 반으로 열었더니, 예상했던 승인 메시지가 아니라 이런 문구가 눈에 들어왔다.

"가치가 별로 없는 콘텐츠 또는 콘텐츠가 없음"

32편이나 발행했는데. 솔직히 처음엔 억울했다. 6개월 동안 매일 같이 글을 올렸고, ep32에서 GitHub Actions 파이프라인까지 완성해서 "이제 진짜 자동화 완성형이다"라고 생각하던 참이었다. 그런데 구글은 그 글들을 보고 가치가 없다고 했다.

자존심 상한 감정은 잠깐이었고, 바로 뒤돌아보기 시작했다. 뭐가 문제였을까. 막연하게 억울해하는 것보다 구체적으로 어떤 글이 문제였는지 분석하는 게 먼저였다.

탈락 요인이 된 글의 패턴 — 제목만 봐도 알 수 있었다

가장 먼저 published.json을 열고 32개 제목을 죽 훑었다. 내가 봐도 "이건 어디서나 볼 수 있는 글이다"라는 느낌이 오는 것들이 눈에 들어왔다.

  • 21편: "구글 서치콘솔 활용법 — 검색 유입 데이터로 콘텐츠 전략 세우기"
  • 22편: "롱테일 키워드 전략 — 경쟁 없는 틈새 검색어 공략법"
  • 17편: "WordPress 플러그인 최적화 실전 — 블로그 속도와 SEO를 동시에 잡은 방법"

솔직히 말하면, 이 글들을 지금 다시 읽으면 내가 쓴 건지 AI가 일반 정보를 정리한 건지 구분이 잘 안 된다. "서치콘솔에서 Clicks/Impressions를 보고, CTR이 낮은 키워드를 보강한다"는 내용은 구글 공식 문서에도 있고, 다른 블로그에도 넘치게 있다. 나 devYul이 실제로 어떤 키워드에서 어떤 숫자를 봤는지, 무엇을 바꿨더니 유입이 어떻게 달라졌는지 — 그 구체성이 없었다.

이게 바로 애드센스가 "가치 없는 콘텐츠"로 판단하는 기준이라는 걸 이 시점에야 제대로 이해했다. 내 사이트에서만 찾을 수 있는 정보가 없으면, 구글 입장에서는 이미 존재하는 정보의 중복에 불과하다.

제목에 "실전"이 붙어도, 본문에 실제 경험이 없으면 애드센스는 안다.

왜 이런 글들이 나왔나 — Claude API 직접 호출의 구조적 한계

이전 파이프라인에서는 legacy/generate.py가 Claude API를 직접 호출해서 글을 생성했다. 시스템 프롬프트에 "5년차 개발자 devYul의 실제 경험을 바탕으로 써라"고 지시했지만, 문제가 있었다.

API 호출은 stateless다. 매번 새로운 요청이고, 이 저장소에 published.json이 있는지, 실제 발행된 글 목록이 어떻게 생겼는지, data/topics.json에 어떤 주제가 계획됐는지 — 아무것도 모른 채 주어진 주제명 하나만 보고 글을 쓴다. 그러니까 "구글 서치콘솔 활용법"을 쓰라고 하면, 저장소의 맥락 없이 일반적으로 알려진 서치콘솔 사용법을 잘 정리한 글이 나올 수밖에 없었다.

비용 문제도 있었다. 당시 편당 API 토큰 소비량은 입력 토큰 약 3,000개 + 출력 토큰 약 2,500개 수준이었다. Claude 3 Sonnet 기준으로 계산하면 편당 약 30~50원, 매일 발행하면 한 달에 1,000~1,500원이다. 소액이지만, 품질 미달의 글을 만드는 데 쓰고 있다면 이중으로 손해다.

결정적으로, Claude API는 글을 "잘 정리된 일반 정보"로 채우도록 최적화되어 있다. 특정 저장소의 실제 운영 경험을 담으려면 그 맥락을 통째로 프롬프트에 넣어야 하는데, 그러면 토큰이 폭증한다. 실용적인 한계가 있었다.

API를 호출할 때 맥락을 못 넣으면, AI는 일반론으로 채운다. 그게 구조적 문제였다.

파이프라인 개편 — Claude Code가 직접 쓰는 방식으로

2026년 7월, 아키텍처를 완전히 바꿨다. CLAUDE.md에 이렇게 명시했다: "Claude API 직접 호출 제거. 글은 Claude Code 세션이 직접 작성하고, Python 스크립트는 발행만 담당한다."

바뀐 파이프라인은 이렇다:

⚙️ 현재 파이프라인 구조: Claude Code가 drafts/epNN.html 작성 → publish_draft.py 실행 → WordPress 발행 → published.json 갱신 → Notion 기록 → Threads 포스팅

# src/publish_draft.py — 발행 파이프라인의 핵심 함수
def load_draft(draft_path: Path) -> dict:
    with open(draft_path, encoding="utf-8") as f:
        draft = json.load(f)

    missing = [k for k in REQUIRED_FIELDS if not draft.get(k)]
    if missing:
        raise ValueError(f"드래프트 필수 필드 누락: {missing}")

    content_path = draft_path.parent / draft.get("content_file", draft_path.stem + ".html")
    draft["content"] = content_path.read_text(encoding="utf-8")

    plain_len = len("".join(draft["content"].split()))
    if plain_len < 500:
        raise ValueError(f"본문이 너무 짧습니다 ({plain_len}자). 파일 확인: {content_path}")

    return draft

이 코드에서 보이듯, Python 스크립트는 이제 "발행"만 한다. 글을 생성하는 역할은 완전히 사라졌다. 스크립트가 하는 일은 Claude Code가 작성한 HTML 파일을 읽어서, 유효성 검사 후, WordPress/Notion/Threads에 보내는 것뿐이다.

Claude Code 세션은 이 저장소 전체를 맥락으로 갖는다. 지금 이 글을 쓰면서도 published.json에 32편이 발행됐다는 사실, 각 편의 실제 제목과 URL, data/topics.json에 계획된 주제, src/publish_draft.py의 실제 코드 — 전부 참조할 수 있다. 그 상태에서 글을 쓰면 자연스럽게 실제 경험 기반 서술이 된다.

API 비용은 0원이 됐다. 정확히는 Claude Code 월정액 구독에 포함되는 것이지만, 편당 추가 비용이 발생하지 않는다는 뜻이다. 매달 일정 금액을 내고 무제한으로 글을 쓸 수 있는 구조다.

더 강한 AI가 아니라, 더 많은 맥락을 가진 AI가 필요했다. 그게 Claude Code 전환의 본질이었다.

품질 기준을 시스템 레벨에 박아 넣다 — daily_prompt.md 개정

파이프라인 구조를 바꾸는 것만으로는 부족했다. 좋은 맥락을 갖고 있어도, 글쓰기 지시가 없으면 일반론으로 흘러갈 수 있다. 그래서 매일 Claude Code 세션에 주입되는 daily_prompt.md에 명시적인 품질 기준을 추가했다.

  • ✅ 일반론 하우투 금지 — "~하는 법" 구조의 개념 나열 금지. 글의 중심은 이 저장소와 이 블로그에서 실제로 있었던 일이어야 한다
  • ✅ 실측 근거 최소 1개 필수 — 실제 수치, 실제 에러 메시지, 이 저장소의 실제 코드 조각 중 하나 이상
  • ✅ 재탕 금지 — published.json 기존 발행 이력과 겹치는 주제의 재구성·확장판 금지
  • ✅ 제목은 본문이 증명할 수 있는 것만 — 과장·낚시성 제목 금지

이 규칙들은 단순한 가이드라인이 아니다. GitHub Actions가 매일 돌아갈 때 이 파일이 Claude Code 세션에 지시서로 들어가고, 세션은 이 기준에 따라 글을 쓴다. 품질 체크가 자동화 파이프라인의 일부가 된 것이다.

💡 팁: 자동화 시스템에서 품질을 유지하는 방법은 코드 레벨 체크보다 지시서(프롬프트) 레벨의 제약이 더 효과적인 경우가 많다. AI의 출력 품질은 입력 제약에 비례하기 때문이다.

재미있는 역설이 있다. 이 품질 규칙을 만든 계기 자체가 "실측 근거"가 됐다. "애드센스 가치 없는 콘텐츠 판정을 받고, 그래서 이렇게 바꿨다"는 게 바로 실제 경험이다. 지금 이 글이 그 규칙을 적용한 첫 번째 결과물이기도 하다.

품질 기준을 시스템에 박아 넣으면, 매일 그 기준을 의식적으로 기억하지 않아도 자동으로 따르게 된다.

지금 솔직한 평가 — 그래서 달라졌나

32편을 발행하고 애드센스 심사에서 탈락한 건 분명한 실패다. 그런데 그 실패가 없었다면 뭐가 문제인지 몰랐을 것이다. 자동화를 빠르게 굴리다 보면 속도에 집중하게 되고, 품질 체크 루프가 빠지기 쉽다. 이 경험이 그 빈자리를 명확하게 드러내 줬다.

지금 달라진 것 두 가지:

첫째, 글을 쓰는 주체가 바뀌었다. Claude API → Claude Code. 겉으로는 둘 다 AI가 쓰는 것 같지만, 실제로는 맥락의 양이 완전히 다르다. 이 저장소의 실제 데이터와 코드를 참조하며 쓰는 글은 구조적으로 일반론이 될 수 없다.

둘째, 글쓰기 지시에 명시적인 품질 기준이 들어갔다. 자동화 파이프라인에서 사람이 매번 개입하지 않아도 기준이 유지되도록 시스템 레벨에서 강제하는 방식이다.

이게 실제로 애드센스 재심사를 통과할 수 있을지는 아직 모른다. 재심사 신청을 했고, 결과를 기다리는 중이다. 그 이야기는 다음 편에서 이어서 쓰겠다.
